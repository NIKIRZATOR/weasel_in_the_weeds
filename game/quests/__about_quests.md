# Каталог `game/quests`

Здесь находится минимальная система квестов игры.

Она не хранит отдельный сложный runtime-state, а работает поверх уже существующих `story_flags` игрока.

Это важно:
- мир и сюжет по-прежнему управляются через `Player.story_flags`;
- квестовая система только интерпретирует эти флаги как задания для игрока;
- благодаря этому не дублируется логика прогрессии.

## Структура

- `models.py` — модели квестов и целей.
- `catalog.py` — список квестов и их порядок.
- `manager.py` — вычисление статусов, активного задания, следующего задания и данных для журнала.
- `__init__.py` — экспорт публичных сущностей.

## Основные сущности

### `QuestObjectiveDefinition`

Описывает одну цель квеста.

Поля:
- `kind` — тип цели.
- `text_key` — ключ локализации текста цели.
- `flag` — флаг, который должен быть получен для завершения цели.
- `item_id` — предмет, который нужен для цели типа `item`.
- `quantity` — количество предметов для цели типа `item`.

Сейчас поддерживаются:
- `flag`
- `item`

На практике в текущем проекте используются в основном `flag`-цели.

### `QuestDefinition`

Описывает сам квест.

Поля:
- `id` — внутренний идентификатор.
- `title_key` — ключ локализации названия.
- `description_key` — ключ локализации описания.
- `category` — категория, сейчас используется `main`.
- `sort_order` — порядок в сюжетной цепочке.
- `required_flags` — флаги, которые должны быть у игрока, чтобы квест стал активным.
- `objectives` — список целей квеста.

## Где задается порядок квестов

Порядок задается в `catalog.py`.

Именно список `QUEST_DEFINITIONS` вместе с `sort_order` определяет:
- какой квест идет раньше;
- какой считается текущим;
- какой считается следующим;
- в каком порядке задания показываются в журнале.

То есть да: каталог квестов и есть порядок основной цепочки.

## Как определяется статус квеста

Это делает `QuestManager.get_quest_status(...)`.

Статусы:
- `inactive` — квест еще не доступен игроку;
- `active` — квест уже доступен и не завершен;
- `completed` — все цели квеста выполнены.

Логика:
1. Если все `objectives` завершены, квест считается `completed`.
2. Если квест не завершен, но выполнены `required_flags`, он считается `active`.
3. Иначе он считается `inactive`.

## Как определяется текущее и следующее задание

### Текущее задание

`QuestManager.get_active_main_quest()`

Берет первый `main`-квест, который имеет статус `active`.

### Следующее задание

`QuestManager.get_next_main_quest()`

Берет следующий `main`-квест после активного, который еще не завершен.

Если активного квеста нет, система берет первый еще не завершенный квест из цепочки.

## Что показывается игроку

### HUD

В HUD отображается только текущее активное задание.

Это делается через:
- `QuestManager.get_active_main_quest()`
- `HUD.draw_active_quest(...)`

### Журнал заданий

Отдельная сцена:
- `game/scenes/quest_log_scene.py`

В журнале показываются:
- выполненные задания;
- текущее активное;
- следующее.

Для списка используется:
- `QuestManager.get_quest_log_entries(...)`

Правила:
- `completed` — показывается с зеленой рамкой;
- `active` — показывается как текущее задание;
- `next` — показывается как следующее.

Игрок может:
- прокручивать список колесом;
- выбирать задание мышкой;
- переключать выбор клавишами `W/S` или `Up/Down`.

При открытии сцены по умолчанию выбирается квест со статусом `active`.

## Откуда квесты берут прогресс

Источник истины — игрок:
- `Player.story_flags`
- инвентарь игрока
- квестовый инвентарь игрока

Это значит:
- диалог может выставить флаг;
- переход уровня может выставить флаг;
- убийство босса может выставить флаг;
- после этого квесты автоматически обновят свой статус.

## Связь с другими системами

### Диалоги

Диалоги уже умеют выдавать награды и ставить флаги.

Пример:
- разговор с мышью-отшельником выставляет `met_mouse_hermit`;
- после этого квест на разговор считается завершенным.

### Переходы уровней

Переходы уже умеют выставлять `set_flags`.

Пример:
- вход в новую локацию может выставить `entered_forest_edge`;
- соответствующий квест после этого завершится автоматически.

### Босс

При смерти босса `EnemyManager` выставляет флаг поражения босса:
- `boss_defeated`

После этого квест на победу над боссом считается завершенным.

## Почему система сделана именно так

Причины:
- мало дублирования;
- быстро расширяется;
- легко отлаживать;
- уже встроена в текущую архитектуру без большого рефакторинга.

Если бы квесты хранили отдельный параллельный state, пришлось бы синхронизировать:
- диалоги;
- переходы;
- предметы;
- босса;
- сюжетные флаги.

Сейчас это не нужно.

## Как добавить новый квест

1. Добавить новый `QuestDefinition` в `catalog.py`.
2. Задать:
- `id`
- `title_key`
- `description_key`
- `sort_order`
- `required_flags`
- `objectives`
3. Добавить тексты локализации в:
- `game/localization/translations/ru.json`
- `game/localization/translations/en.json`
4. Убедиться, что нужный флаг или предмет реально появляется в игре.

## Пример

Пример логики:

- квест `meet_hermit`
- цель: получить флаг `met_mouse_hermit`
- флаг ставится в диалоге
- после разговора:
- цель считается завершенной;
- квест становится `completed`;
- следующий квест в цепочке может стать `active`

## Ограничения текущей версии

Сейчас система:
- не хранит отдельный save-state квестов;
- не имеет side quests;
- не имеет наград на уровне квеста;
- не имеет ручного принятия / сдачи квеста;
- не имеет сложных счетчиков вроде `убей 10 врагов`.

Это сознательно упрощенная версия под текущий вертикальный срез игры.
